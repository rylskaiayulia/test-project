# Тестовое задание — Junior System Analyst

## Задание 1. Анализ требований

### Раздел ТЗ: Функционал корзины

### 1. Логические недочеты и противоречия в предоставленном ТЗ

#### Пункт 1

> Пользователь может добавить в корзину от 1 до 10 единиц одного товара.

Логически противоречит пунктам 3 и 4, так как суммарное количество не может превышать 20 штук (п.4), а в корзине может находиться не более 5 различных товаров (п.3). Получается ограничение на количество единиц одного товара требует уточнения.

---

#### Пункт 2

> Пользователь может изменить количество каждого товара в корзине не менее, чем до 1-го. Для удаления товара используется отдельная кнопка.

Противоречит пункту 9 (автоматическое удаление товара при уменьшении количества до 0).

---

#### Пункт 3

> В корзине может находиться не более 5 различных товаров.

Требуется уточнение совместимости ограничения с количеством единиц одного товара и общим лимитом.

---

#### Пункт 4

> Суммарное количество всех товаров в корзине не может превышать 20 штук.

Требуется уточнение взаимодействия ограничения с количеством различных товаров и ограничением на единицы одного товара.

---

#### Пункт 6

> При попытке добавить товар, превышающий лимиты, показывается сообщение: «Лимит корзины превышен».

Требуется уточнить:

* превышение количества одного товара;
* превышение количества разных товаров;
* превышение общего количества товаров.

---

#### Пункт 7

> Цена на продукт фиксируется на момент добавления и не меняется.

Конфликтует с пунктом 13 (автоматическое обновление цены в корзине).

Требуется уточнить:

* цена фиксируется навсегда;
* цена обновляется при изменении каталога.

---

#### Пункт 9

> Если пользователь уменьшает количество товара до 0, товар удаляется из корзины.

Противоречит пункту 2.

---

#### Пункт 10

> В корзине может быть реклама других продуктов.

Требуется уточнить:

* количество рекламных блоков;
* алгоритм отображения;
* условия показа.

---

#### Пункт 11

> Реклама товаров должна отображаться каждый будний день утром и вечером.

Требуется уточнить механизм реализации и временные интервалы.

---

#### Пункт 13

> Если цена изменилась в каталоге, система должна автоматически обновить её в корзине.

Противоречит пункту 7.

Требуется определить единое правило формирования цены.

---

## Ниже предлагаемая версия требований

* Пользователь может добавлять в корзину до 5 различных товаров.
* Максимальное общее количество товаров в корзине — 50 единиц. Для каждого товара можно указать количество от 1 до 10 единиц.
* После добавления товара цена фиксируется на момент добавления, но если цена в каталоге меняется, система автоматически обновляет цену в корзине для всех пользователей.
* Пользователь может уменьшить количество товара до 1. Если уменьшение до 0, товар удаляется из корзины.
* При попытке добавить товар, превышающий лимиты (по количеству разных товаров или общему количеству товаров), система показывает сообщение «Лимит корзины превышен».
* В корзине отображается список товаров, их количество, цена за единицу и итоговая сумма по каждой позиции.
* Визуализировать рекламу товаров в корзине по заданному алгоритму (алгоритм требует уточнения).

# Задание 2. Проектирование REST API

## Сценарий работы

1. Пользователь открывает экран «Выберите магазин».
2. Приложение вызывает API.
3. Сервер возвращает список магазинов.
4. Приложение отображает карточки.
5. При нажатии выполняется переход по внешней ссылке.

# Задание 3. Архитектура PUSH-уведомлений

## Верхнеуровневая схема

```text
Микросервисы
(Cart / Order / Marketing)
          ↓
Message Broker
(Kafka / RabbitMQ)
          ↓
Notification Service
          ↓
Push Provider
(FCM / APNs)
          ↓
Мобильное приложение
```

---

## Компоненты

### Cart Service / Order Service / Marketing Service

Создают события:

* корзина долго не изменялась;
* заказ отменён;
* стартовала рассылка.

---

### Message Broker

Функции:

* принимает события;
* хранит;
* передаёт подписчикам.

Пример события:

```json
{
  "eventType": "CART_ABANDONED",
  "userId": 123,
  "cartId": 555
}
```

---

### Notification Service

Функции:

* получает событие;
* определяет шаблон;
* получает push-токен;
* отправляет PUSH.

Пример уведомления:

```json
{
  "title": "Корзина ждёт вас",
  "body": "Завершите оформление заказа"
}
```

---

### Push Provider

Доставка уведомлений:

* Android → FCM
* iOS → APNs

---

## Сценарий работы

1. Пользователь оставил товары в корзине.
2. Cart Service обнаружил отсутствие активности.
3. Сервис публикует событие.
4. Notification Service получает событие.
5. Формирует уведомление.
6. Отправляет через Push Provider.
7. Пользователь получает PUSH.
